Claude holds commit authority on your GitHub corpus. ChatGPT and Gemini don't — but they can still read your context if Claude packages it for them first. This guide explains the handoff pattern and how to trigger it.

Context Packager Guide

Covers how to send work to Gemini or ChatGPT with full context, bring outputs back into Claude, and keep GitHub as the single source of truth across platforms.

Read guide

Paste into Claude and say: "Load this and set up my cross-platform workflow."

Why This Exists

Different AI platforms have different strengths. Claude handles strategic writing, document creation, and GitHub commits. Gemini has native Google Workspace integration, making it fast for Gmail-heavy tasks and Google Docs. ChatGPT has certain coding and browsing strengths depending on the task. You shouldn't have to pick one and stay there.

The problem is context. Your GitHub corpus — your profile, your task context, your project background — lives in a private repo. Other platforms can't read it natively. Every time you switch, you either lose your context or spend time re-explaining it.

The solution is packaging. Claude reads your repo, bundles the relevant context into a self-contained prompt, and hands it off. You paste it into the other platform. The other platform does the work. You bring the output back to Claude, which updates the context file and commits everything to GitHub. Claude stays as the hub; the other platforms are task-specific spokes.

Platform Roles

Platform Role Best for
Claude Hub — reads and commits to GitHub Strategic writing, document creation, task management, stakeholder communication
Gemini Spoke — read-only via packaged prompt Gmail-integrated workflows, Google Docs drafts, OKR development, grant documentation
ChatGPT Spoke — can connect directly to GitHub for reading Task-specific co-work, visual design iterations, coding tasks
Perplexity Research spoke Citation-heavy research, competitive analysis, industry context

The Handoff Pattern

Sending work to Gemini

Say to Claude: "Prepare a Gemini session for task [#]."

Claude will package a self-contained prompt containing:

  • Your condensed AI profile (communication preferences, tone, collaboration mode)
  • The full task context entry from task_context.md
  • Any relevant reference material from the repo
  • Working instructions for Gemini, including what to produce and how to format outputs for bringing back to Claude

You copy the packaged prompt, open Gemini, paste it in, and work. When you're done, copy the output and bring it back to Claude: "Here's the Gemini output for task [#]. Update the context file and commit."

Sending work to ChatGPT

ChatGPT can connect to your GitHub repo directly via its GitHub integration (Settings → Connected apps → GitHub). Once connected, it can read files from your repo without needing a packaged prompt.

Still say to Claude first: "Prepare a ChatGPT session for task [#]." Claude will produce a shorter handoff note — the task context summary and instructions — since ChatGPT can read the repo itself for background.

When you bring outputs back, same pattern: give them to Claude to update the context file and commit.

The return flow matters

This only works if outputs come back to Claude. If you do good work in Gemini and never bring it back, the context file goes stale and GitHub stops being the source of truth. The habit is: anything that moves a task forward gets committed. Claude handles that step — you just need to paste the output in and ask it to update.

Setting Up the Trigger

Add this to your AI interaction profile under collaboration mode or notes:

cross_platform:
  packager_trigger: "Prepare a [platform] session for task [#]"
  return_trigger: "Here's the [platform] output for task [#]"
  commit_authority: "Claude only"
  platforms:
    gemini:
      access: "Packaged prompt — no direct repo access"
      best_for: "Gmail workflows, Google Docs, OKR development"
    chatgpt:
      access: "GitHub OAuth connector (read-only)"
      best_for: "Task co-work, coding, visual iteration"
    perplexity:
      access: "Read-only via packaged context"
      best_for: "Research, citations, competitive analysis"

What a Packaged Prompt Looks Like

When you say "Prepare a Gemini session for task 12," Claude produces something like this — a fully self-contained prompt you paste directly into Gemini:

## Context for this session

You are helping [Name] with a specific task. Here is the context you need.

### About [Name]
[Condensed profile: role, org, communication preferences, tone guidance, things to avoid]

### Task: [Task Name]
[Full context entry from task_context.md: what it is, audience, constraints, success criteria, background]

### Your job in this session
[Specific instruction: what to produce, format, length, what to flag]

### When you're done
Produce your output clearly labeled so it can be copied and brought back to Claude
for context file update and GitHub commit. Note any assumptions you made.

The packaged prompt carries everything the other platform needs. It doesn't require the model to have any prior context — each session is self-contained.

When to Use Which Platform

Use Gemini when

  • The task involves Gmail deeply — drafting replies, processing a thread, summarizing an inbox
  • You're working in Google Docs and want inline assistance
  • Grant reporting that benefits from Google Workspace integration
  • OKR development where you want to export directly into Sheets

Use ChatGPT when

  • You need to iterate on visual or design outputs (ChatGPT has DALL-E built in)
  • The task involves significant code and you want browser/code execution
  • You want to read directly from your GitHub repo without a packaged prompt

Use Perplexity when

  • You need cited, sourced research — competitive analysis, industry context, regulatory background
  • You want real-time web search with citations rather than Claude's training data
  • The output is research that Claude will then use to draft something

Keep it in Claude when

  • Strategic writing, stakeholder communication, executive-level documents
  • Anything that needs to be committed to GitHub
  • Task management and session work
  • Anything where the context is rich enough that a packaged prompt would lose nuance

Keeping GitHub as Source of Truth

The system breaks down if outputs from other platforms don't come back. Three rules that keep it working:

  1. Always return outputs to Claude. Even if the work happened in Gemini, Claude is the one that updates the context file and commits. The other platform never writes to the repo.
  2. Commit after every meaningful session. If work moved forward — a draft was produced, a decision was made, context was developed — it gets committed. Not at the end of the week. At the end of the session.
  3. Context file is the handoff layer. The context entry in task_context.md is what carries work between sessions and platforms. Keep it current. When you bring outputs back to Claude, ask it to update the working notes in the context entry before committing.
