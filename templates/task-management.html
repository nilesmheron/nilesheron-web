Task Management Setup Guide

Sets up your task dashboard, context file, inbox, and session workflow. Covers the two-line task format, co-work vs. solo classification, Granola note routing, and session start/end protocol. Requires a GitHub corpus already set up.

Read guide

Paste into Claude and say: "Load this and set up my task management system."

What This System Is

Your GitHub repo already stores your AI profile. This extends it into a live task management system — a task_dashboard.md file that Claude reads at the start of every session, updates throughout, and commits back at the end. GitHub is the source of truth. Claude is the interface.

Inputs flow in from wherever you already work: Slack, Gmail, Granola meeting notes, or just talking to Claude directly. At the start of each session, Claude checks all of them, surfaces anything new, and shows you the full dashboard before asking what you want to work on.

What you'll have by the end

  • A task_dashboard.md file structured for Claude to read and update
  • A task_context.md file with working briefs for co-work tasks
  • A session startup trigger phrase Claude responds to
  • An inbox section for quick mobile captures
  • Connected inputs from whichever of Slack, Gmail, and Granola you use

Step 1 — Create the Dashboard File

In your [username]-ai-management repo, navigate to the tasks/ folder and create task_dashboard.md. Paste the template below as your starting point — Claude will help you populate it.

Dashboard template

# [Your Name]'s Task Dashboard
*Last Updated: [DATE]*

---

## SYSTEM INSTRUCTIONS

### Trigger phrase
When the user says "Start task session," execute the session startup sequence below.

### Session startup sequence
1. Read this file from GitHub
2. Process any items in the INBOX section — convert to proper tasks, ask clarifying questions, clear the inbox
3. Move tasks in RECENTLY COMPLETED older than 72 hours to COMPLETED ARCHIVE
4. Check Gmail ([your-email], label: claude_tasks) for new forwarded tasks
5. Check Slack ([#your-inbox-channel]) for new messages
6. Query Granola for action items from the last 10 days
7. Display the current dashboard
8. Ask: "What do you want to work on?"

### Session end
1. Commit updated dashboard and task_context.md to GitHub
2. Post session summary to Slack [#your-inbox-channel]
3. Confirm both completed

### Task format (two lines)
Line 1: **[#]. [Task Name]** | [Priority: High/Med/Low] | [Timeline]
Line 2: Co-work or Solo | [RP: Name — if applicable]

### Co-work vs. Solo
- Co-work: Tasks you and Claude work on together. Always have an entry in task_context.md.
- Solo: Tasks you handle independently. Dashboard only — no context entry needed.

### Inbox
Drop raw task captures here anytime (especially from mobile). Claude formats and files them at session start.
Format: Task name | Project | Priority (optional) | Deadline (optional)

### Completed tasks
- Completed within 72 hours → RECENTLY COMPLETED (visible)
- Older than 72 hours → COMPLETED ARCHIVE (stored, not displayed)
- Format: **Task name** (Completed YYYY-MM-DD)

### GitHub is source of truth
Always read the latest file from GitHub at session start.
Never recreate — only read and update.
Always commit at session end.

Step 2 — Create the Context File

task_context.md holds working briefs for co-work tasks — the background, constraints, audience, and success criteria Claude needs before it can do meaningful work on a task. You don't fill it out for every task, only the ones you'll actually work on with Claude.

How it works

When you say "Work on task [#]" or "Pull context for [Task Name]," Claude reads this file and finds the entry. The status field tells it what to do next:

  • Pre-filled — context is ready, Claude reads and begins
  • Needs interview — Claude runs an intake interview before any work starts
  • Partial — Claude reviews what exists, fills gaps, then begins

Context file template

# Task Context File
*Working briefs for co-work tasks. Pull on demand.*

---

## [#]. [Task Name]
**Status**: Needs interview / Pre-filled / Partial
**Last updated**: [DATE]

**What it is**: [1-2 sentence description]
**Audience**: [Who will see or use this output]
**Constraints**: [Tone, length, format, things to avoid]
**Success criteria**: [What done looks like]
**Background**: [Anything Claude needs to know — stakeholder context, prior work, sensitivities]
**Working notes**: [Evolving notes from sessions]

---

When to add a context entry

Add one when you're about to work on something. You don't need to fill out every field before starting — status "Needs interview" tells Claude to ask you what it needs. The context file grows as you work, not before.

Step 3 — Set Your Trigger Phrase

Add the following to your userPreferences in Claude.ai (Settings → Profile), below your profile load instructions:

Task management trigger: "Start task session"
When I say this, read tasks/task_dashboard.md from GitHub and run the session startup sequence defined in the file's SYSTEM INSTRUCTIONS.

That's it. From now on, saying "Start task session" in any Claude conversation kicks off the full sequence — GitHub read, inbox processing, integrations check, dashboard display.

Step 4 — Organize Your Projects

The dashboard organizes tasks by project or organization, then by priority within each. There's no fixed list — use whatever project groupings match your actual work. Niles uses: TechTown Detroit, Hyde Park, Verse and Hook, LCR Detroit, Other, Personal.

Within each project: High Priority, Medium Priority, Low Priority / Ongoing. Tasks move up and down as things shift — Claude handles the re-sorting when you tell it a priority has changed.

Step 5 — Connect Your Inputs

The system pulls new tasks from three sources at session start. Connect whichever ones you use.

Gmail

Create a label called claude_tasks in Gmail. When you want Claude to action an email — a follow-up you need to track, a commitment you made on a call — apply that label. Claude checks it at session start and suggests converting flagged emails into tasks.

Slack inbox channel

Your #[firstname]-claude-inbox channel serves double duty: it's where you drop quick task captures throughout the day, and where Claude posts session summaries at the end. Drop anything there — a task name, a note, a link — and Claude will process it into a proper task at your next session.

Granola

Claude queries Granola for the last 10 days of meeting notes at session start and surfaces anything that looks like an action item. It'll suggest adding them to the dashboard — you approve or skip each one. Meeting notes can also be committed to tasks/notes/[project]/ for reference.

Running a Session

Starting

Say: "Start task session." Claude reads GitHub, processes your inbox, checks integrations, and shows you the dashboard. Then: "What do you want to work on?"

Working on a task

Say: "Work on task 4" or "Pull context for [task name]." Claude loads the context entry and begins. If there's no context entry yet, it runs a short intake interview first.

Adding tasks mid-session

Just say it: "Add a task — follow up with [person] about [thing], medium priority, TechTown." Claude formats it and adds it to the dashboard immediately.

Completing tasks

"Mark task 7 complete." Claude moves it to Recently Completed with today's date.

Ending

When you're done, Claude commits the updated dashboard and context file to GitHub and posts a summary to your Slack inbox channel. If you forget to end formally, you can say "Wrap up the session" at any point.

The Two-Line Task Format

Every task follows the same two-line structure. This is what makes the dashboard machine-readable as well as human-readable.

**[#]. [Task Name]** | [Priority] | [Timeline]
Co-work or Solo | [RP: Name]

Examples:

**1. Q2 Investor Update Deck** | High | Due Apr 15
Co-work

**2. Send contract to vendor** | Med | ASAP
Solo | [RP: Jordan]

**3. Annual report draft** | High | May
Co-work | [RP: Sam]

The responsible party ([RP: Name]) marks who owns execution — useful when you're tracking delegated work. Timeline uses plain language: "ASAP," "This week," "Apr 15," "TBD," "May."

Granola Note Routing

When Claude surfaces a Granola meeting note, it analyzes the content, suggests which project folder it belongs in, and asks for your approval before committing:

tasks/notes/[project]/YYYY-MM-DD-meeting-title.md

You say yes, or redirect it to a different folder. Claude commits. The notes are then available as reference material in future sessions when you're working on related tasks.

Profile Update — Adding Task Management

Add this block to your AI interaction profile under integrations, so Claude always knows where your task files live:

task_management:
  dashboard: "tasks/task_dashboard.md"
  context: "tasks/task_context.md"
  trigger: "Start task session"
  inbox_channel: "#[firstname]-claude-inbox"
  gmail_label: "claude_tasks"
  granola: true
  commit_authority: "Claude only"
  note: "GitHub is source of truth. Never recreate files — read and update only."

System Health and Maintenance

The bottom of your dashboard has a System Health section — a quick count of active tasks by project, how many are urgent, how many are co-work vs. solo. Claude updates this automatically at session end.

A few maintenance habits worth keeping:

  • If the dashboard hasn't been updated in more than a week, start a session just to process the inbox and re-sync — stale dashboards drift fast
  • Context entries should be updated after each working session, not just before — "Working notes" in the context file is where the live thinking goes
  • The completed archive is searchable: "What did I complete in March?" works as a query
  • Tasks that have been sitting on the dashboard for more than a month with no movement are either low-priority-forever or should be deleted — Claude can help with a periodic cleanup pass
